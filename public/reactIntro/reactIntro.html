Single Page Applications

Before we start focusing on React, you need to understand the concept Single Page Application (SPA)

A SPA is an application that runs inside a browser and does not require page reloading during use. Think of a SPA as a traditional Desktop Client loaded from a web server.

A SPA is not a boring application with only one page. It loads its static code as "one page" and dynamically update that page, typically via a REST backend, when the user interacts with the page

Single Page Applications Continued

Traditional Web apps, versus Single Page Application

In a traditional Web app, every time the app calls the server, the server renders a new HTML page. This triggers a page refresh in the browser. This is what you did on 2. semester using Java JSP.

In an SPA, after the first page loads, all interaction with the server happens through AJAX calls. These AJAX calls return data—not markup—usually in JSON format. The app uses the JSON data to update the page dynamically, without reloading the page

Single Page Applications Pros & Cons

The Pros

  • SPA's is (sometimes) faster, since most resources (HTML+CSS+JS) are only loaded once throughout the lifespan of application. Only data is transmitted back and forth.
  • Applications are more fluid and responsive, without the flickering effect of reloading and re-rendering the page
  • Another advantage comes from the fact, that they often use REST back-ends. With REST we can reuse the same backend for Web, Mobile, Desktop, IOT etc.
  • Rendering is done by clients. Saves the Server from rendering "millions" of pages.
  • SPA's can cache data locally. An application sends only one request, store all data, then it can use this data and works even offline
  • Easy to separate work among backend and frontend developers (REST)

Single Page Applications Pros & Cons

The CONS

  • It is more complex to make SEO optimization of a Single-Page Application, since content is loaded asynchronously.
  • Initial download can be slow, because:
    • Client frameworks (React etc.) are required to be loaded to the client
    • All static Code must be loaded upfront (for large applications, static code can be divided up into smaller chunks, which are only loaded when required, making it multipage, single page app ;-)
  • It requires JavaScript to be present and enabled. If a user disables JavaScript, it won’t be possible to present the application in a correct way.

Babel, Webpack and Node

React requires (for realistic React Development) ES6 (and sometimes 7..) which older browsers do not understand and also JSX, which NO browsers understand

Because of this, React apps must be pre-compiled (transpiled is the corrrect term) into ES5. This is done with the help of two tools:

  • Babel - the transpiler that transpiles ES6+JSX into ES5
  • Webpack - the "build tool" we use to activate babel and to combine all our static resources into one file (SPA)
  • Node and npm Node makes it possible to run JavaScript on our computer (required for the tools above)and npm is what we will use to install required dependencies (similar to how Maven handles Java dependencies)

React - The Virtual Dom

Manipulating the DOM is a fundamental part of all modern, interactive web. DOM manipulation unfortunately, is much slower than plain JS-operations

React's solution to this problem, is a Virtual Dom. For every DOM object, there is a corresponding "virtual DOM object." A virtual DOM object is a representation of a DOM object, like a lightweight copy.

  • The entire virtual DOM gets updated
  • The virtual DOM gets compared to what it looked like before you updated it. React figures out which objects have changed
  • The changed objects, and the changed objects only, get updated on the real DOM
  • Changes on the real DOM cause the screen to change
